# HTML Screenshot Skill - 更新日志

## v2.1 - Channel-Specific Rules (2026-02-07)

### 🎯 针对 Telegram 优化

**原因**：用户观察到 Telegram 比其他渠道更依赖截图

### 渠道差异化

| 渠道 | 触发规则 | 原因 |
|------|---------|------|
| **Telegram** | **任何视觉变化**（更主动） | 用户看不到本地文件 |
| 浏览器 | 重要视觉变化 | 用户可以自己刷新预览 |
| Terminal | 重要视觉变化 | 用户可以 `open` 文件 |

### 新增判断

- Telegram: 微调也截图（如 margin 10px→12px）
- 浏览器/Terminal: 微调不截图（用户自己看）

## v2.0 - Simplified (2026-02-07)

### 🎯 重大简化

**原因**：用户反思 + Opus 深度分析发现过度设计

### 核心变化

| 维度 | v1 → v2 |
|------|---------|
| **触发条件** | 任何 HTML 修改 → **只有视觉修改** |
| **截图频率** | 100% → **~60%**（减少 40% 噪音） |
| **用户控制** | 无 → **可触发/跳过** |
| **错误处理** | 默默失败 → **明确告知** |

### 新增功能

1. **智能判断**：区分视觉修改 vs 代码重构
2. **用户控制**：
   - "看看效果" → 强制截图
   - "不用看" → 跳过本次
3. **错误处理**：
   - 截图失败 → 告知原因 + 文件路径
   - 询问是否重试
   - Never fail silently

### 删除的过度设计

- ❌ 自动重试机制（太复杂）
- ❌ 迭代检测（不可靠）
- ❌ 批量智能合并（YAGNI）
- ❌ 复杂的视觉重要性评分（简化为规则）

### 保留的核心

- ✅ agent-browser 截图（免费、准确）
- ✅ 渠道智能适配（Telegram/浏览器/终端）
- ✅ 完全本地化（无 API 成本）

## 关键洞察（Opus 分析）

### 场景分析
- 10 个真实场景中，只有 **6 个需要截图**
- 40% 的截图是无意义的（注释、格式化、重构）

### 工具对比
- Figma、VS Code、CodePen 都是**让用户主动看**
- 我们是**推送模式**（Telegram 特性）
- 需要更克制

### 80/20 原则
- 核心功能（20%）：视觉修改截图 + 用户控制 + 错误告知
- 过度设计（80%）：复杂判断逻辑、自动重试、批量合并

## 迁移指南（v1 → v2）

### 行为变化

**v1**：
```
修改任何代码 → 自动截图
```

**v2**：
```
修改视觉代码 → 自动截图
修改逻辑代码 → 不截图（除非用户说"看看效果"）
```

### 新增交互

**用户现在可以**：
- 说"看看效果" → 强制截图
- 说"不用看" → 跳过截图

**失败时**：
- v1：默默失败
- v2：告知原因 + 文件路径 + 询问重试

---

## 反思

### 为什么过度设计？

1. **假设错误**：以为用户总是想看效果
2. **缺乏验证**：没有真实使用场景
3. **追求完美**：试图处理所有边界情况

### 简化的价值

1. **减少噪音**：40% 无用截图消失
2. **用户掌控**：可以说"不用看"
3. **更可靠**：失败时明确告知
4. **更易维护**：代码更简单
