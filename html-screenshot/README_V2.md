# HTML Screenshot Skill (Simplified v2)

**一句话**：视觉修改完成后自动截图，失败时告知用户，用户随时可控制。

---

## 🎯 为什么简化？

**v1 的问题**（过度设计）：
- ❌ 任何修改都截图（包括注释、格式化）
- ❌ 无法区分视觉修改 vs 代码重构
- ❌ 失败时默默忽略
- ❌ 用户无法控制

**v2 的改进**（简化+智能）：
- ✅ **只对视觉修改截图**（60% 场景，减少噪音）
- ✅ **用户可控**（"看看效果" / "不用看"）
- ✅ **失败时明确告知**（不默默失败）

---

## 🔧 核心规则（渠道差异化）

### Telegram vs 浏览器/Terminal

| 修改类型 | Telegram | 浏览器/Terminal | 原因 |
|---------|----------|----------------|------|
| **样式变化** | ✅ 任何变化都截 | ✅ 重要变化才截 | Telegram 用户看不到本地文件 |
| **微调** (margin 10→12px) | ✅ 截图 | ❌ 不截 | Telegram 更主动 |
| **文本改动** | ✅ 任何文本 | ❌ 重要文本 | Telegram 依赖截图 |
| **布局变化** | ✅ 任何布局 | ✅ 重要布局 | 都需要 |
| **新增元素** | ✅ 截图 | ✅ 截图 | 都需要 |
| **代码重构** | ❌ 不截 | ❌ 不截 | 都不需要 |
| **注释/格式化** | ❌ 不截 | ❌ 不截 | 都不需要 |

### 为什么区分渠道？

**Telegram**：
- 用户**看不到本地文件**
- **必须靠截图**才能看效果
- 更主动截图（宁可多不可少）

**浏览器/Terminal**：
- 用户**可以自己预览**
- 截图是**辅助**，不是必需
- 只截重要变化（减少打扰）

## 🎮 用户控制

| 你说 | 我做 |
|------|------|
| "看看效果" | 立即截图（即使不是视觉修改） |
| "不用看" | 跳过本次截图 |
| （什么都不说） | 视觉修改 = 自动截图 |

---

## ❌ 错误处理（新增）

**原则：Never fail silently**

截图失败时：
1. ✅ 明确告知失败原因
2. ✅ 提供文件路径（用户可手动打开）
3. ✅ 询问是否重试

**示例**：
```
❌ 截图失败：Browser not available
📁 你可以手动打开：/path/to/file.html
🔄 需要我重试吗？
```

## 🎓 关键学习

### Opus 分析发现
1. **60/40 法则**：只有 60% 修改需要截图
2. **其他工具都是让用户主动看**（Figma、VS Code）
3. **推送 vs 拉取**：Telegram 场景特殊（聊天 = 推送）
4. **失败必告知**："保证能发出来" = 失败时明确告知

### 简化原则（80/20）
- 保留核心功能：视觉修改截图
- 删除过度设计：复杂判断、自动重试
- 增加用户控制：可触发、可跳过
- 增加容错：失败时告知 + 备选方案
